# 底层软件项目周计划（满强度压缩版）

## 一、使用说明

这份文档只服务一个目标：把预热阶段、前两个核心项目，以及后面的 PCIe 亮点项目拆成可以逐周推进、逐周验收、逐周复盘的满强度执行计划。

这里的阶段定义如下：

1. 预热阶段：裸机 AXI DMA 速通
2. 第一个项目：Linux 驱动与 BSP 基础项目群
3. 第二个项目：Linux DMA + 中断 + 持续数据流主项目
4. 第三个项目：PCIe + QEMU 亮点项目

执行原则：

1. 每周都要有代码产出，不接受纯看资料周。
2. 每周都要有文档产出，不接受只写代码不总结。
3. 每周都要有验证动作，不接受“感觉差不多会了”。
4. 每周结束必须回答三个问题：做成了什么、卡在哪里、下周怎么补。
5. 这份周计划默认按高负荷执行，适合当前希望先把强度拉满、尽快完成第一轮技术闭环的节奏。

## 二、预热阶段周计划

## 阶段定位

预热阶段不是核心卖点，它的作用是用最短时间建立 DMA、寄存器、缓存一致性的底层直觉，避免后面的 Linux DMA 项目变成空学。

建议周期：2 周

### 第 1 周：裸机 DMA 最小闭环

本周目标：跑通一次最小裸机 DMA 数据搬运闭环。

本周任务：

1. 明确 FPGA 侧数据源和 ARM 侧接收路径。
2. 梳理 AXI4-Lite、AXI4-Stream、AXI DMA 的基本关系。
3. ARM 端用裸机 C 配置 DMA 寄存器。
4. 跑通一次数据从 FPGA 到 ARM 内存的搬运。
5. 用打印或校验确认数据结果正确。

本周交付：

1. 一个最小裸机 DMA Demo。
2. 一份寄存器控制流程笔记。
3. 一张最小数据搬运流程图。

本周验收标准：

1. 能说明数据是怎么搬过去的。
2. 能指出关键寄存器大致在控制什么。
3. 能证明一次闭环已经跑通。

### 第 2 周：缓存一致性和迁移视角

本周目标：把裸机 DMA 的结果整理成后续 Linux 项目的预备知识。

本周任务：

1. 总结缓存一致性的最基础概念。
2. 梳理物理地址、寄存器配置、传输触发之间的关系。
3. 写一份“裸机 DMA 到 Linux DMA 将增加哪些复杂度”的总结。
4. 整理 3 分钟口述稿。
5. 复盘当前裸机版本的边界。

本周交付：

1. 1 份缓存一致性和地址关系笔记。
2. 1 份“裸机到 Linux 的迁移差异”总结。
3. 1 份 3 分钟口述稿。
4. 1 份阶段复盘。

本周验收标准：

1. 能讲清为什么裸机 DMA 不是最终求职卖点。
2. 能说明迁移到 Linux 后会新增哪些问题。
3. 能把这个阶段控制在预热范围内，不继续恋战。

## 三、第一个项目周计划

## 项目定位

这个项目不是为了炫技，而是为了给后面的 DMA 主项目和 PCIe 项目打地基。

最终目标不是“会写最小字符驱动”，而是形成一组带工程味的 Linux 驱动基础材料：

1. 一个字符设备驱动
2. 一个常见外设驱动案例
3. 一套设备树、platform driver、日志调试、用户态测试的基础能力

建议周期：7 周

### 第 1 周：环境和最小驱动框架

本周目标：把 Linux 驱动开发环境、内核模块编译加载和最小字符驱动骨架全部搭起来。

本周任务：

1. 配好交叉编译环境或本地内核模块编译环境。
2. 跑通最小 hello module，确认 insmod、rmmod、dmesg 路径正常。
3. 建一个字符设备驱动工程目录，写出最小模块框架。
4. 理清 cdev、major/minor、device node 的基本关系。
5. 记录模块加载、卸载、打印日志的标准流程。

本周交付：

1. 一个可编译、可加载、可卸载的最小模块。
2. 一份环境搭建记录。
3. 一份“模块生命周期”笔记。

本周验收标准：

1. 能独立解释 module_init 和 module_exit 在干什么。
2. 能通过 dmesg 确认模块加载成功。
3. 能说清字符设备驱动后面还要补哪些部分。

### 第 2 周：字符设备驱动主路径

本周目标：做出可正常 open、read、write 的字符设备驱动主路径。

本周任务：

1. 写 file_operations 基础接口。
2. 完成设备号申请、cdev 注册、class 和 device 创建设备节点。
3. 写一个简单的内核缓冲区，支持用户态读写。
4. 写配套用户态测试程序。
5. 跑通 open、read、write 整条链路。

本周交付：

1. 一个可正常读写的字符设备驱动。
2. 一个用户态测试程序。
3. 一份读写链路流程图。

本周验收标准：

1. 驱动加载后可以看到设备节点。
2. 用户态程序能写入并读回数据。
3. 能讲清从用户态到内核态发生了什么。

### 第 3 周：ioctl、错误处理、工程味补强

本周目标：把字符设备驱动从“能跑”升级到“有工程味”。

本周任务：

1. 增加 ioctl 控制接口。
2. 给 read、write、ioctl 增加基础错误处理。
3. 增加参数合法性检查和日志打印。
4. 把用户态测试程序扩成多命令测试工具。
5. 总结常见错误路径。

本周交付：

1. 一个支持 ioctl 的字符设备驱动。
2. 一个支持多种测试动作的用户态工具。
3. 一份错误处理清单。

本周验收标准：

1. 能解释为什么很多驱动需要 ioctl 而不只是 read/write。
2. 能演示至少一种错误路径。
3. 能说明当前驱动距离真实业务驱动还差什么。

### 第 4 周：设备树和 platform driver 入门

本周目标：把“字符设备驱动”视角切到“平台驱动”视角。

本周任务：

1. 学 compatible、reg、interrupts 等常见设备树属性。
2. 写一个最小设备树节点。
3. 写最小 platform driver，跑通 probe 和 remove。
4. 在 probe 里拿到资源并打印。
5. 总结 platform driver 和字符设备驱动的关系。

本周交付：

1. 一个最小 platform driver 样例。
2. 一份最小设备树节点示例。
3. 一份 probe 流程笔记。

本周验收标准：

1. 能讲清 compatible 为什么重要。
2. 能说明 probe 为什么会触发。
3. 能讲清 platform driver 和字符设备驱动分别解决什么问题。

### 第 5 周：常见外设驱动案例

本周目标：完成一个带 Linux 框架味道的 SPI、I2C 或 UART 驱动案例。

本周任务：

1. 从 SPI、I2C、UART 中选一个最适合当前环境展示的方向。
2. 不重复深挖协议原理，重点放在 Linux 驱动框架。
3. 结合设备树和 platform 匹配写出最小驱动路径。
4. 写一个基础通信或状态读写例子。
5. 形成“FPGA 手搓协议 vs Linux 驱动框架”的对照总结。

本周交付：

1. 一个常见外设驱动案例。
2. 一份用户态验证脚本或测试程序。
3. 一份软硬件对照总结。

本周验收标准：

1. 能讲清为什么你在 FPGA 里会协议，不等于你会 Linux 外设驱动。
2. 能讲清 Linux 框架帮你做了哪些事。
3. 能把这个外设驱动讲成一个有价值的辅助项目，而不是凑数项目。

### 第 6 周：中断、等待队列、调试能力补强

本周目标：把驱动能力从同步路径提升到带事件响应的路径。

本周任务：

1. 学中断注册、处理中断、释放中断的基本流程。
2. 至少在一个小驱动里加入中断或等待队列机制。
3. 理解阻塞与非阻塞思路。
4. 记录 dmesg、设备节点、设备树、寄存器排查方法。
5. 把前面两个辅助项目做一次问题定位复盘。

本周交付：

1. 一个包含中断或等待队列实践的小驱动版本。
2. 一份常见调试路径总结。
3. 一份问题复盘文档。

本周验收标准：

1. 能讲清中断相比轮询的意义。
2. 能说出等待队列的基本应用场景。
3. 面试里能回答“你怎么调驱动问题”的基础问题。

### 第 7 周：整体验收与简历化

本周目标：把前 6 周的内容收束成可用于后续主项目和面试的资产。

本周任务：

1. 整理字符设备驱动项目。
2. 整理常见外设驱动项目。
3. 输出 Linux 驱动基础笔记。
4. 输出启动链路总结。
5. 写第一个项目的 2 分钟版和 5 分钟版讲稿。

本周交付：

1. 字符设备驱动项目材料包。
2. 外设驱动项目材料包。
3. Linux 驱动基础笔记。
4. 启动链路总结。
5. 2 分钟和 5 分钟版项目讲稿。

本周验收标准：

1. 能把这个阶段讲成“Linux 驱动/BSP 地基建设”，而不是零散学习记录。
2. 能说清这个阶段如何为 DMA 主项目服务。
3. 能拿出可以直接复用到简历里的文字和图。

## 四、第二个项目周计划

## 项目定位

这是整套路线里最值钱的主项目，也是最应该优先投入精力的硬骨头。

最终目标不是“做出一个 DMA demo”，而是做出一个具备下面特征的项目：

1. Linux 平台驱动
2. DMA + 中断
3. 持续数据流
4. 用户态工具链
5. 错误处理和恢复能力
6. 吞吐、延迟或稳定性测试

建议周期：12 周

### 第 1 周：主项目框架和资源梳理

本周目标：把项目架构、硬件资源、驱动骨架先搭起来。

本周任务：

1. 明确 FPGA 侧数据源和 ARM 侧数据接收路径。
2. 写主项目架构图。
3. 列出寄存器、DMA、IRQ、设备树节点这些核心资源。
4. 搭建 Linux 平台驱动骨架。
5. 跑通最小 probe 和 remove。

本周交付：

1. 主项目架构图。
2. 一份资源清单。
3. 一个最小平台驱动骨架。

本周验收标准：

1. 能说明整个系统的数据流方向。
2. 能讲清项目里的硬件资源有哪些。
3. 能确认 probe 已经正确跑起来。

### 第 2 周：设备树、资源获取、寄存器映射

本周目标：把设备树、寄存器和资源获取打通。

本周任务：

1. 写主项目设备树节点。
2. 在驱动中获取内存资源和中断资源。
3. 完成寄存器映射。
4. 用寄存器读写确认设备访问路径正常。
5. 记录设备树和资源获取的关键代码路径。

本周交付：

1. 主项目设备树节点。
2. 能正常映射寄存器并进行基本访问的驱动版本。
3. 一份资源获取流程记录。

本周验收标准：

1. 能解释资源是如何从设备树进入驱动的。
2. 能确认寄存器访问路径正确。
3. 能说明资源拿错时会出现什么现象。

### 第 3 周：DMA buffer 与单次传输

本周目标：先打通单次 DMA 传输，形成项目最小闭环。

本周任务：

1. 理解 dma_alloc_coherent 的使用场景。
2. 申请 DMA buffer。
3. 配置 DMA 相关寄存器。
4. 完成单次数据传输。
5. 在用户态或内核日志中看到正确结果。

本周交付：

1. 一个能完成单次 DMA 传输的版本。
2. 一份 DMA buffer 使用说明。
3. 一份单次传输时序图。

本周验收标准：

1. 能解释为什么这里需要 DMA。
2. 能说明 buffer、地址、缓存一致性之间的关系。
3. 能证明单次传输已经跑通。

### 第 4 周：中断驱动链路

本周目标：从轮询切到中断通知，提升项目真实感。

本周任务：

1. 申请中断并注册处理函数。
2. 在 DMA 完成后通过中断通知驱动。
3. 用日志记录中断响应路径。
4. 替换掉单纯轮询的完成判断。
5. 整理中断时序流程。

本周交付：

1. 一个中断驱动版本。
2. 一份中断响应流程图。
3. 一份“轮询 vs 中断”对比笔记。

本周验收标准：

1. 能解释中断为什么比轮询更像真实驱动场景。
2. 能确认中断完整触发和处理。
3. 能说明中断链路里的易错点。

### 第 5 周：用户态接口和测试工具

本周目标：把主项目从“内核里跑通”升级成“用户态能用”。

本周任务：

1. 设计主项目的用户态接口。
2. 实现初始化、触发传输、读取结果、结果校验等基础控制路径。
3. 支持至少一种阻塞读取方式。
4. 写用户态测试工具。
5. 确认用户态到驱动到硬件再返回的闭环。

本周交付：

1. 一个用户态测试工具。
2. 一份接口设计说明。
3. 一份端到端闭环说明。

本周验收标准：

1. 用户态能稳定触发并拿到数据。
2. 能讲清用户态和驱动之间如何协作。
3. 项目不再只是“内核 demo”，而是具备基本可用性。

### 第 6 周：持续数据流和 buffer 复用

本周目标：把项目从单次 DMA demo 升级成持续流项目。

本周任务：

1. 设计持续数据流传输路径。
2. 实现 buffer 复用、双缓冲或 ping-pong 中的一种。
3. 确认多次传输不会立刻出错。
4. 记录状态切换和同步点。
5. 输出持续流版本的时序图。

本周交付：

1. 一个支持持续传输的版本。
2. 一份 buffer 复用机制说明。
3. 一份持续流时序图。

本周验收标准：

1. 能连续完成多次传输。
2. 能解释为什么持续流比单次 demo 更有求职价值。
3. 能说明 buffer 复用设计的基本思路。

### 第 7 周：错误处理和恢复机制

本周目标：补齐项目里最像工程项目的那一层。

本周任务：

1. 加入超时处理。
2. 加入异常状态检测。
3. 至少实现一种恢复机制，例如链路重启或缓冲区重置。
4. 模拟一类故障并记录定位过程。
5. 把错误路径整理成可讲材料。

本周交付：

1. 一个具备基础错误处理的版本。
2. 一份错误路径说明。
3. 一份故障定位案例。

本周验收标准：

1. 能讲清项目出错时怎么办。
2. 能说出恢复机制是怎么设计的。
3. 项目从 demo 更接近真实驱动工程。

### 第 8 周：性能测试和稳定性测试

本周目标：让项目具备“数据说话”的部分。

本周任务：

1. 设计吞吐测试方案。
2. 设计延迟或 CPU 占用测试方案。
3. 设计稳定性测试方案，例如长时间连续传输。
4. 收集测试数据并整理图表。
5. 分析当前瓶颈。

本周交付：

1. 一份测试方案。
2. 一份测试数据记录。
3. 一份测试图表或表格。

本周验收标准：

1. 至少完成两个维度的测试。
2. 能讲清测试结果意味着什么。
3. 能指出项目当前性能瓶颈在哪里。

### 第 9 周：调试案例集整理

本周目标：把踩过的坑变成面试里的高价值内容。

本周任务：

1. 整理地址映射相关问题。
2. 整理 DMA buffer 相关问题。

### 第 10 周：项目材料和讲稿初版

本周目标：把主项目收束成简历资产。

本周任务：

1. 写主项目 10 分钟版讲稿。

本周目标：针对前 10 周暴露的短板做最后一轮加强。

本周任务：

## 五、第三个项目周计划

这是用来拉高简历辨识度和技术上限的亮点项目。

最终目标不是“看过 PCIe”，而是做出一个可以写进简历、能在面试中稳定讲清框架、而且带有最小设备驱动闭环的 PCIe 项目。

建议周期：8 周

### 第 1 周：PCIe 最小理论框架

本周目标：只补驱动必须知道的 PCIe 基础，不陷进协议深坑。

本周任务：

1. 梳理 PCIe 拓扑和枚举的最基础概念。
2. 理解 BAR、配置空间、设备发现的基本作用。
3. 理解 PCIe 驱动和平台驱动最核心的差异。
